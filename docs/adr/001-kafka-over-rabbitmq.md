# ADR-001: Apache Kafka as the central event backbone

**Status:** Accepted  
**Date:** 2024-02-15  
**Author:** Platform Architecture, SRE, Ingestion Team  
**Superseded by:** None

## Context

The AI Document Compliance Platform must process regulatory submissions asynchronously: ingestion, OCR, classification, fraud checks, LLM reasoning, and human review orchestration. Each step is owned by a different team and must scale independently. We needed a message broker that could handle backpressure during traffic spikes, support multiple consumer groups reading the same events, and allow replay of historical data when models or rules change.

Initial prototypes (2023) used RabbitMQ, but we experienced repeated operational issues under the load profile generated by corporate filing deadlines (see INC-2023-11-07 and INC-2024-01-22). RabbitMQ’s memory-based delivery, combined with the required consumer acknowledgement patterns, led to head-of-line blocking when a single consumer was slow (e.g., OCR pod under memory pressure).

## Decision

We standardise on **Apache Kafka** as the sole asynchronous communication layer between services. Every domain event is published to a dedicated topic. Services declare their consumed topics and group IDs. The platform SDK (`compliance-sdk`) provides a `ResilientConsumer` abstraction that implements per‑message retries and dead‑letter queue routing.

## Consequences

- **Decoupled deployability:** Services can be deployed, scaled, and upgraded without coordination; a new version of the fraud engine can reprocess past events from the log.
- **Replayability:** Model retraining or rule changes can be applied retrospectively by rewinding consumer offsets.
- **Operational complexity:** The team must manage ZooKeeper (or KRaft), topic creation, partition reassignment, and schema evolution. We rely on Confluent’s `cp-kafka` in Kubernetes and have SRE runbooks for partition rebalance storms.
- **Ordering guarantees:** Within a partition, ordering is preserved; we key messages by `document_id` to ensure a single document’s events are processed sequentially by any consuming service that cares about order (e.g., the compliance engine must see `classified` before `ocr.completed` can be evaluated).
- **Long retention:** 7‑day default retention on all topics; audit‑critical topics (`document.ingested`) are retained for 30 days to allow reprocessing.

## Tradeoffs

- **Latency vs throughput:** End‑to‑end processing latency increased from ~500ms (RabbitMQ) to ~2–5 seconds in the common case because consumers poll on a 500ms interval. This is acceptable for regulatory filings that are not user‑facing; the intake API still returns a `received` status within 200ms.
- **Message size limits:** Kafka’s default 1MB message size forced us to store raw document bytes in MinIO and publish only metadata. This added indirection but kept brokers performant.
- **Consumer lag visibility:** We built custom Prometheus alerting (`kafka_consumer_lag_records`) because the built‑in JMX metrics did not expose per‑partition lag in a way that matched our service ownership model.

## Alternatives Considered

- **RabbitMQ:** Better for complex routing, but we hit memory‑based backpressure during the Q1 2024 filing spike where OCR pods crashed; RabbitMQ acknowledged messages to slow consumers caused queue growth that evicted other services’ messages. Its lack of a built‑in replay log meant we would need a separate event store for reprocessing.
- **AWS SQS/SNS:** Not self‑hosted; regulator data sovereignty requirements mandate on‑prem or air‑gapped cloud, ruling out managed services.
- **NATS Streaming:** Evaluated but lacked mature ecosystem (Kafka Connect, KSQL) for audit‑log integration.
- **Direct HTTP/gRPC orchestration:** Rejected because it couples services, requires circuit breakers everywhere, and cannot replay events.

## Operational Impact

- **Incident response:** The SRE team can use `kafka-consumer-groups` to inspect lag and reset offsets. During INC-2025-08-14, the on‑call engineer manually paused the `document.ocr.requested` consumer to let the compliance engine catch up, then resumed.
- **Scaling:** When the OCR pipeline is saturated, we scale consumer pods; Kafka automatically redistributes partitions. However, we discovered that with >50 partitions the rebalance time exceeds 30s, leading to temporary consumer pauses. We now cap at 24 partitions per topic and scale via replicas.
- **Monitoring:** We expose consumer lag as a Prometheus gauge. A Grafana dashboard panel shows lag per topic; critical topics alert at 5,000 lag.
- **Security:** Topics are not encrypted at rest (on‑prem disks are encrypted at storage level); in‑transit encryption is achieved via mTLS between brokers and clients, configured through the SDK’s Kafka client.

## Migration Path

Legacy RabbitMQ exchanges (`document.uploaded`, `ocr.finished`) were bridged to Kafka via a temporary connector (Jan–Mar 2024). The connector was decommissioned in April 2024 after all services migrated. No production services depend on RabbitMQ now.
